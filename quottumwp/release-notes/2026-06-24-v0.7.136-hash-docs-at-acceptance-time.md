## Acceptance log now stores content hash

When a user accepts the ToS/Privacy, we now fetch the marketing site's published `/version.json` and record the SHA-256 of the exact doc bytes alongside the URL and version. URL + version + hash = the precise wording the user agreed to, which closes the proof loop for any later dispute.

Best-effort: if the manifest fetch fails, acceptance still records — the hash field stays null rather than blocking signup over a proof-of-content field.

## Cross-repo PR workflow (in `gaviso/quottum-site`)

The marketing site repo now auto-opens a PR here whenever `terms.astro` or `privacy.astro` changes on `main`, bumping `TOS_VERSION` to the new "Last updated" date. The PR description has a materiality checklist — merge to force everyone to re-accept, or close to leave `TOS_VERSION` unchanged for typo/link fixes. No more forgetting to bump after a real change; no more diluting consent with a meaningless re-prompt for a fix.

Setup needed once: create a fine-grained PAT scoped to this repo (Contents + Pull requests, read+write), add as `SAAS_PR_TOKEN` in the site repo's Actions secrets.
