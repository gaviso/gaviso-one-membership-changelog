Three UX changes, two of them via paired background agents with a reviewed, gated merge.

### Billing — inline plan-picker accordion

"View plans" on **/dashboard/settings/billing** now expands the plan picker **in place** instead of navigating to /dashboard/upgrade — so "what you have" → "what's locked" → pick a plan all happen on one page.

Built the right way (anti-drift): a new shared `resolveUpgradeFormProps(workspaceId, role)` helper is the **single source** for the picker's props + self-serve eligibility + the billable-plans catalog. **Both** the upgrade page and the billing accordion call it. The new `InlinePlanPicker` reuses the **existing** `UpgradeForm` component (not a rebuild) in a collapsible region; the "Unlock more with an upgrade" link opens the same picker via a window event (one picker, one open-state). The upgrade page was refactored onto the helper with **zero behavior change** (banners, already-subscribed branch, the current-plan CTA fix, trial flows, credit, URL-derived cadence/plan all preserved).

Selecting a paid plan still routes to Stripe Checkout — the accordion just removes the intermediate page hop.

### Clients — Classification + Brand-logo pane on the new-client page

The new-client page had the modern form but dropped the edit page's right pane. Added it (Brand-logo stub + Classification: Stage / Industry / Client since) — **and made it persist**: `POST /api/clients` previously inserted only company/contacts/notes and would have **silently dropped** stage/industry/client_since. It now accepts and inserts them (stage validated to active|lead|paused|lost with an 'active' fallback; blanks → null).

Extracted a shared `ClientSidePane` used by both new + edit pages (same anti-drift pattern as the shared `ClientForm`); the edit page's behavior is unchanged. (Removed a now-dead `f()` handler during review.)

### Settings — sub-nav active style

The settings sub-nav active item now matches the **main nav's** active recipe exactly (`--brand-tint` background + `--accent` text, with the light-mode contrast override), reusing the same tokens so the two stay in sync.

### Process

The two larger changes were built by parallel background agents on non-overlapping files, then **reviewed before merge** — confirmed both refactors were behavior-preserving, the create-API actually persists the new fields, and a dead handler was cleaned up — with a final unified type-check on the composed tree. No migration; pure code.
