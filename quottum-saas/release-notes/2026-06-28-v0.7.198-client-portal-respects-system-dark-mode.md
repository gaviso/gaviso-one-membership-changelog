The token-gated \`/p/[token]\` client portal hardcoded \`data-theme="light"\` and baked in \`#faf6ee\` cream + white card colors, forcing light mode regardless of the visitor's system preference. Visitors on dark-mode systems got a jarring flash of bright cream.

### Fix

- Removed the hardcoded \`data-theme="light"\` and \`colorScheme:'light'\` override on the portal's outer div. The page now inherits the root layout's \`html[data-theme=...]\` attribute, which the existing pre-paint init script sets from \`prefers-color-scheme: dark\`.

- Added \`--portal-canvas\` CSS variable to \`src/styles/ds/colors.css\` — \`#faf6ee\` in light mode (the deliberate warm cream brand feel), \`#161616\` in dark mode (the standard dark canvas). Tokenized this specifically because the cream is intentional brand identity, not just an off-the-shelf surface color.

- Swapped all literal colors to existing theme tokens: \`bg-[#faf6ee]\` → \`bg-[var(--portal-canvas)]\`, \`bg-white\` → \`bg-[var(--surface-card)]\`, button \`text-white\` → \`text-[var(--surface-card)]\` (inverts cleanly with \`--ink-900\` in both themes).

### Six surfaces updated

Outer canvas, header card, awaiting-approval banner, empty-state card, quote-list articles, contracts table card, and the "View quote" CTA button.

### Out of scope

\`/q/[token]\` (public quote view) and \`/c/[token]\` (public contract view) still force light theme. Those are intentionally document-like ("looks like paper") — switching them to dark is a separate UX decision and we left them alone.

### No flicker

The pre-paint init script in the root layout runs before first paint, sets \`html[data-theme]\` from \`prefers-color-scheme: dark\`, then the portal renders straight into the correct theme. No flash, no client-side toggle needed (visitors aren't authenticated, no \`quottum_theme\` preference to read).

Pure code; no migration.
