Two changes: pagination for all activity cards, and a hotfix for a v0.7.201 regression caught by post-release verification.

### Activity/timeline pagination (user + admin)

The activity cards rendered their entire event list. Now they paginate using the existing `usePagination` + `TablePagination` infra — **10 per page**, with 10/25/50 options.

- **User-side** `ActivityTimeline` (quote / contract / client detail) — replaced the old "show more" expand cap with real pagination.
- **Admin** `WorkspaceTimeline` (/admin/workspaces/[id]) — the card that showed 20+ entries flat now paginates. Kept it a server component (it does heavy data assembly); only the list rendering was extracted into a new `'use client'` `WorkspaceTimelineList`.
- `TablePagination` gained an optional `options` prop (defaults to the existing 25/50/100), so the five existing table consumers are completely unaffected.

### Branded-email logo hotfix

v0.7.201 shipped `automated_emails` — Pro/Business workspaces get their own logo in outbound emails, served from `GET /api/branding/logo/[workspaceId]` (decodes the stored data-URI into real image bytes so it renders in Gmail/Outlook, which strip data-URI `<img>`).

**The bug:** the auth middleware (`proxy.ts`) didn't allowlist that route, so unauthenticated requests — every email client — got a **401** and the logo broke.

**Caught by the post-release verification agent.** Code-presence checks passed (the route is correct), but the live curl returned 401 — exactly the runtime gap a static audit can't see. Added a tight allowlist entry scoped to the `[workspaceId]` leaf.

**Verified live post-deploy:**
- Workspace with a logo → `200`, `image/png`, real image bytes ✓
- Workspace without → `404` (handler reached) ✓
- Both were `401` before the fix.

Near-zero real-world impact (branded emails are brand-new, nobody relied on them yet) — fixed before anyone does.

### No migration

Pure code. Type-checked clean on the composed tree.
