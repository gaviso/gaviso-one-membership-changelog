Security fix for the audit's A1 finding.

The "first-workspace-only" gate in `/api/auth/signup-workspace` and `provisionBetaSignup` was a check-then-insert race — two concurrent POSTs from one fresh user both passed the "no workspace yet" check and each created a Pro-trialing workspace, and since `trial_used_at` is per-workspace, each got a **fresh trial**.

Fix: a `SECURITY DEFINER` RPC `create_first_workspace()` takes a per-user advisory lock, re-checks the gate, and inserts the workspace + owner membership + eager settings row in **one transaction**. Returns the new id, or NULL when a workspace already exists. Atomic (no orphan on failure), service-role-only. Migration 093.

Verified: the gate returns NULL and creates nothing for an existing user.
