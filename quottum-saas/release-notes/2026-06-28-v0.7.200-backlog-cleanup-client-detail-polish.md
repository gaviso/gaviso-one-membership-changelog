Five fixes batched — closes three items from the v0.7.196 follow-up backlog plus two new client-detail UI requests.

### Impersonation timezone guard

The v0.7.195 browser-TZ default leaked when an operator created a workspace WHILE impersonating a customer — the request body carried the operator's browser zone, not the customer's. Both creation endpoints (`/api/auth/signup-workspace`, `/api/workspaces`) now reject the body timezone when the `IMPERSONATE_COOKIE` is set, falling through to the `UTC` column default. Owner can correct via Settings → General.

### Block emails + popups show expiry

When an operator blocks a feature with an expiry date, the customer email and in-app popup now render "This change is in effect until <date>." — symmetric with the grant-added `until_summary` that already existed. Forever blocks render no expiry line (no trailing-punctuation artifact). Migration 088 updates the `workspace.feature_blocked` template.

### Multi-event popup stacking

Previously the welcome-popup slot was one-at-a-time: if an operator fired several feature changes in quick succession (grant A, block B, remove C), only the first showed a popup — the rest dropped to email-only. Now the 2nd+ event APPENDS its section to the pending popup (thin divider between), with an `events[]` audit array stored in the jsonb. Acknowledging dismisses the whole stack. Combined with v0.7.195's live poller, a user idle on the dashboard sees ONE stacked popup with all pending events within ~30s.

### Client portal controls reflect the gate

The client detail page now computes `hasFeature(wsId, 'client_portal')` and passes it to the share controls. When the feature is blocked or not on plan, the "Client portal" pill renders disabled (greyed, tooltip) instead of the live link + Copy + Rotate — so the owner can't hand out a link that lands on the v0.7.199 blocked portal view. The `/p/[token]` page enforces independently; this is the matching UI affordance.

### Removed redundant Email button

Dropped the duplicate "Email" mailto pill from the client detail header — the email is already a mailto link in the contact-info block below.
