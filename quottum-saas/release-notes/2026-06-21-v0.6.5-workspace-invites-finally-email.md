The longest-standing TODO in the codebase: workspace invites have been creating tokens since v0.4.5-a but never actually sending email. The comment in the handler said \"v0.6.0 will do it\" — well, v0.6.5 does it.

### What changed

POST \`/api/workspaces/invitations\` now fires a branded invite email immediately after creating the invitation row. Recipient gets:

> **{{inviter}} invited you to {{workspace}} on Quottum**
>
> Click below to accept — takes about 30 seconds.
> [Accept invite]

Same visual language as the rest of the Quottum auth emails: sprout mark, green CTA, copy-paste fallback URL, short \"What's Quottum?\" pitch for first-time recipients.

### Variables exposed to the template

| Var | Source |
|---|---|
| \`{{workspace_name}}\` | current workspace |
| \`{{invited_by_name}}\` | inviter's \`user_metadata.full_name\` / \`.name\` (falls back to email) |
| \`{{role}}\` | \`owner\` or \`member\` |
| \`{{invite_link}}\` | \`\${NEXT_PUBLIC_APP_URL}/invite/{token}\` |
| \`{{recipient_email}}\` | for the safety footer |

Migration 019 seeds it as a platform default (\`workspace_id IS NULL\` row in \`email_templates\`).

### Failure handling

\`sendEmail()\` never throws. If the configured provider is down or rejects the address, the row lands in \`email_log\` as \`failed\` and the inviter can still copy the token URL from the Settings UI to share manually. No regression vs. the old \"manual share\" flow.
