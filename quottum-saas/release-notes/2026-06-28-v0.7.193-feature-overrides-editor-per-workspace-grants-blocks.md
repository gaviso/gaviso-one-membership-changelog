Step 3 of the operator subscription-actions surface. The last dormant column from migration 077 — \`workspaces.feature_grants\` and \`feature_blocks\` — is no longer dead schema. Operators can grant a Pro user a Business-tier feature without bumping their plan (policy A: feature uplifts go on the workspace, not on the catalog) and can block plan features per-workspace when needed.

### Added

- **\`POST /api/admin/workspaces/[id]/feature-overrides/[kind]\`** — adds or replaces an override.
  - \`kind\` ∈ \`grant\` (force-on) | \`block\` (force-off)
  - Body: \`{ feature_key, until?, reason_category, reason_text? }\`
  - \`feature_key\` validated against the \`features\` catalog (typo guard). Inactive features are allowed for operator-intent reasons; the audit captures the choice.
  - \`until\` validated as future ISO datetime; null/empty means forever.
  - \`reason_category\` required from the G taxonomy; \`reason_text\` optional (≤2000 chars).
  - Logs \`admin_audit\` \`workspace.feature_grant.set\` or \`workspace.feature_block.set\` (\`metadata.replaced=true\` when an entry already existed).
- **\`DELETE /api/admin/workspaces/[id]/feature-overrides/[kind]/[feature_key]\`** — removes an override. 409 \`override_not_found\` if the key wasn't present. Logs \`workspace.feature_grant.cleared\` or \`workspace.feature_block.cleared\`.
- **\`<AdminWorkspaceFeatureOverrides>\`** card on \`/admin/workspaces/[id]\` (inserted right below the Subscription card so it sits in the operator-action cluster).
  - Two columns: **Grants (force-on)** and **Blocks (force-off)**, each with its own \`+ Add\` button.
  - Per-row metadata: feature name + key (mono) · expiry (\`Until X\` or \`Forever\` with infinity icon) · reason_category · "by Operator Name" (resolved via actorNames) · italic reason_text when present · per-row remove button.
  - Compose modal: feature picker (active catalog features) · \`Forever\` checkbox (defaults true; uncheck to set a specific end-of-day expiry in admin TZ) · required reason · optional notes. Warns when replacing an existing override for the same key.

### Changed

- **\`entitlements.getWorkspaceEntitlements\`** — new resolution order:

  \`\`\`
  plan_features (or comp grants minus excluded_features)
    → minus active feature_blocks
    → plus active feature_grants
  \`\`\`

  Each override's \`until\` is checked at query time. Expired entries don't apply even if a future janitor hasn't swept them out of the jsonb yet (correctness is query-time; storage cleanup is later).

### Notes for testing

- New \`Feature overrides\` card visible on \`/admin/workspaces/[id]\` between the Subscription card and the KPIs row.
- Try \`+ Add\` under Grants → pick a feature → keep Forever → reason → Grant. Confirm the row appears with the operator name + reason + remove button.
- Hit a feature-gated endpoint as that workspace's owner — feature should be live.
- Add a Block for the same feature → resolution applies blocks first then grants, so the grant still wins (operator's call; both are stored intentionally).
- Set a future \`until\` date → grant applies until then; expired entries become inactive at query time (the row still sits in the jsonb until manually removed).
