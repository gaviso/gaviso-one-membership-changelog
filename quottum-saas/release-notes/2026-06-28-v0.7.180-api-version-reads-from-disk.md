Eliminates the "deploy ran green but \`/api/version\` still serves old version" failure mode we hit twice in two days (v0.7.177, v0.7.178).

### Added

- **\`/api/version\` reads \`package.json\` from disk at request time** (mtime-cached). The endpoint previously read \`NEXT_PUBLIC_APP_VERSION\` baked into the server bundle, so a successful \`git pull\` + build + failed-restart left it lying. Now the endpoint reports what landed on disk regardless of restart state.
- **Dual-field response**: \`version\` (disk truth — what \`git pull\` landed) and \`bundleVersion\` (baked truth — what the running process is actually serving). Operators can now distinguish "new build on host" from "new build live."

### Fixed

- **\`NewVersionBanner\` gates on \`bundleVersion\`** (with fallback to \`version\` for backward compat). Prevents a refresh-loop edge case where, after the change above, a client running an old bundle would see the disk version, hit reload, and re-receive the old bundle because Enhance hasn't restarted yet.

### Docs

- Committed the parked design doc at \`docs/proposals/api-version-from-disk.md\` as a record of the rationale.
