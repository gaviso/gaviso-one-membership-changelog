Bug-sweep release. Four fixes from the post-v0.5.0 code review — three real and silent.

### Critical: coupon enforcement was effectively a no-op on accept

The public quote-accept route (\`/api/q/[token]\`) runs unauthenticated — the token IS the access control. But \`coupon_redemptions\` and \`coupons\` RLS is members-only. So:

- The redemption INSERT was **silently denied** by RLS → \`single_use\` was never actually enforced
- The recurring-coupon lookup also silently returned null → **recurring coupons never carried onto the auto-spawned contract**

Both bugs only surfaced from a code review (the helpers swallow Supabase errors). Now routed through the service-role client for these specific post-validation writes.

### High: /c/[token] and /p/[token] couldn't show workspace branding

\`workspace_settings\` and \`workspaces\` RLS is members-only too. The anon client returned null for logo / footer / Terms & Conditions / workspace name on both public pages.

Same fix — service-role for these reads, selecting only safe-to-expose fields. (The portal had this gap since v0.4.6 but it was unnoticed because the fallback handled the null case.)

### High: PATCH /api/quotes/[id] couldn't clear a coupon without re-sending items

The coupon block was nested inside \`if (items !== undefined)\`. PATCHing \`{coupon_code: ''}\` alone was a no-op. Refactored: coupon logic runs unconditionally when \`coupon_code\` is in the body.

### Low: proxy /c/* whitelist tightened

\`startsWith('/c/')\` would match any future authenticated subroute. Tightened to a per-token regex.

### Deferred (still queued for the next release)

- Currency threading sweep on dashboard / contracts list / admin metrics (multi-currency workspaces still render rollups as USD)
- Recurring-coupon math on renewal — discount is copied verbatim, so a percent coupon's flat-amount value won't scale with price uplifts. Tracked for v0.6 alongside email.
