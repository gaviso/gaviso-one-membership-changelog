Step 2 of the operator subscription-actions surface. Activates the third dormant column from migration 077 (\`workspaces.welcome_announcement\`). The operator queues a one-shot in-app announcement; the user sees it on next dashboard load and dismisses with **Got it**.

### Added

- **\`POST /api/admin/workspaces/[id]/welcome-popup\`** — operator queues an announcement.
  - Body: \`{ kind, message_html, reason_category, reason_text? }\`
  - kind ∈ \`welcome\` | \`announcement\` | \`feature_grant\`
  - \`message_html\` capped at 10KB so paste-the-whole-page mistakes 400
  - Refuses 409 \`popup_already_queued\` when one is pending — operator clears the old one first rather than silently overwriting
  - Logs \`admin_audit\` \`workspace.welcome_popup.set\`
- **\`DELETE /api/admin/workspaces/[id]/welcome-popup\`** — operator cancels a queued popup before the user reads it. Logs \`workspace.welcome_popup.cleared\`.
- **\`POST /api/me/welcome-popup/acknowledge\`** — any workspace member dismisses. Atomic two-tab safe clear: \`UPDATE … SET welcome_announcement = null WHERE id = ? AND welcome_announcement IS NOT NULL RETURNING\`. \`acknowledged: false\` in the response means another tab beat us to it — UX still closes either way.
- **\`<WelcomeAnnouncement>\`** — full-screen blocking modal on \`/dashboard\`. Single **Got it** CTA per locked policy (no tour). Renders the operator-authored HTML directly (operator is trusted; size is capped server-side).
- **\`<SendWelcomePopupButton>\`** on \`/admin/workspaces/[id]\` — ghost button below the Subscription card. Opens a compose modal with kind dropdown + HTML editor (\`CodeEditor\` from \`/admin/email-templates\`) + reason. When a popup is already queued, swaps to a yellow chip + Cancel button.
- **\`WorkspaceLite.welcome_announcement\`** — surfaced on \`getUserWorkspace\` so the dashboard layout can branch without a second fetch.

### Render precedence (dashboard layout)

The welcome popup is skipped when:

- the operator is impersonating (operator shouldn't trip on the user's popup), or
- the **ReconsentGate** is pending (legal blocking acceptance has priority; the popup waits for the next load).

### Notes for smoke testing

1. \`/admin/workspaces/<id>\` → click **Send welcome popup** → keep defaults → **Queue popup**.
2. Confirm the button is replaced by a yellow **Welcome popup queued · welcome** chip with a Cancel button.
3. Open the workspace's dashboard (fresh tab/incognito) → expect the modal with the default snippet.
4. **Got it** → modal closes; refresh → gone.
5. Back on the admin page → reload → chip gone (column nulled).
6. Two-tab race: open the dashboard in two tabs, click **Got it** in one → second tab's **Got it** should still close cleanly (\`acknowledged: false\` returned, popup goes away regardless).

### Out of scope for this version

- Companion email when popup fires — the locked policy hints at "email also fires" but the trigger isn't clear yet. Held until usage shows it's needed.
- Logging user-side acknowledgments to \`admin_audit\` — operator's \`set\` entry is the audit trail; per-user dismissal would be noise.
