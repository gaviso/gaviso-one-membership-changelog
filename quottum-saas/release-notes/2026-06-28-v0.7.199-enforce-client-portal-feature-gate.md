Audit follow-up #1 from v0.7.197 — fixes a CATALOG-ONLY feature identified by the parallel enforcement audit. Operator-blocked \`client_portal\` now actually refuses \`/p/[token]\` with a respectful "not available" message instead of silently serving the page.

### The bug

\`client_portal\` was a row in the \`features\` table from day one. Operators could grant/block it via the v0.7.193 feature-overrides editor. The block landed in \`workspaces.feature_blocks\`. The resolver in \`entitlements.ts\` filtered it out of the effective set. And then... no caller in the codebase consulted that resolver for this feature. So a blocked client_portal had zero effect — Bobby's portal still loaded.

### Fix

Server-side gate in \`src/app/p/[token]/page.tsx\`, right after the client lookup + token-expiry checks, BEFORE the branding fetch:

\`\`\`tsx
const portalEnabled = await hasFeature(client.workspace_id, 'client_portal')
if (!portalEnabled) return <BlockedPortalView />
\`\`\`

Three deliberate choices:

1. **Gate placed AFTER \`notFound()\`** — invalid / expired tokens still return 404. We don't leak workspace existence by saying "portal disabled" for a bad token.
2. **Blocked view doesn't fetch workspace branding** (logo, name) — avoids a wasteful read AND avoids leaking workspace identity to a visitor who's been explicitly blocked.
3. **Blocked view uses \`--portal-canvas\` + \`--surface-card\`** from v0.7.198 — renders correctly in both light and dark modes. Centered card, respectful copy aimed at the END USER (Margo, not Bobby):

> **This portal isn't available right now**
> The workspace you're trying to reach has disabled access to this client portal. Please reach out to whoever shared the link with you.

### Out of scope

\`/q/[token]\` and \`/c/[token]\` use the quote's / contract's own \`client_token\` (different token, different share semantics) — tied to specific document share links, not "the portal." Treat as separate concern.

### Multi-task protocol validation

This release was the first end-to-end test of the multi-task protocol saved to memory earlier today: dispatched the fix as a background agent in parallel with v0.7.198 push + deploy polling. Agent returned a clean type-checked patch in ~30s while v0.7.198 was still deploying. Zero file conflicts (v0.7.198 was committed first per protocol step "ship task 2 first → fire agent on clean baseline"). Protocol worked exactly as designed.
